The AMR fleet manager architecture builder produces a structured xlsx workbook that specifies the software and communication architecture required to coordinate a fleet of autonomous mobile robots within the operating envelope defined by the upstream zone specification. The architecture covers the VDA 5050 protocol interface between vehicles and the fleet manager, traffic management zone logic, deadlock avoidance strategies, order management workflow, vehicle state machine, and communication redundancy. This workbook provides the architecture-level specification that system integrators use to configure or procure a fleet management system.

    **VDA 5050 protocol overview**

    VDA 5050 is an open protocol developed by the German Association of the Automotive Industry (VDA) and VDMA for standardised AGV and AMR fleet communication. It defines the MQTT message format and topic structure for all fleet manager to vehicle interactions.

    <CardGroup cols={2}>
      <Card title="Order messages" icon="list-ordered">
        Fleet manager sends drive and action orders to the AMR as a sequence of nodes and edges with associated actions. The AMR executes and reports progress via state messages.
      </Card>
      <Card title="State messages" icon="chart-line">
        AMR reports its current position, battery state, load status, active warnings, errors, and order execution progress to the fleet manager at a configurable reporting interval.
      </Card>
      <Card title="instantAction messages" icon="bolt">
        Fleet manager sends immediate commands (e-stop, soft stop, start charging, cancel order) that interrupt normal order execution. Safety-critical path — requires delivery confirmation.
      </Card>
      <Card title="Visualization messages" icon="display">
        AMR reports its position and heading for fleet visualisation display. Not safety-critical — lower QoS acceptable. Used for operator monitoring dashboards.
      </Card>
    </CardGroup>

    **Traffic management zone logic**

    ```text
    Zone entry/exit model (mutex zone pattern):

    1. AMR requests entry to zone Z via instantAction "zoneRequest"
    2. Fleet manager checks zone capacity (max concurrent occupants)
    3. If capacity available: grant entry → AMR proceeds
    4. If at capacity: AMR waits at designated hold point
    5. AMR sends "zoneExit" instantAction on leaving zone Z
    6. Fleet manager decrements zone occupancy, grants entry to next waiting AMR

    Deadlock detection:
    - Monitor all AMR hold queues for circular wait condition
    - Timeout threshold: configurable per zone (default 30 s)
    - Escalation: priority yield command to lowest-priority AMR in cycle
    ```

## Standard reference

**VDA 5050** — *Interface for the communication between automated guided vehicles (AGV) and a master control*

**ISO 3691-4:2020, Section 5.10** — *Control systems*

**ANSI/RIA R15.08-2** — *System-level requirements for industrial mobile robots*

VDA 5050 was developed by VDA and VDMA to enable AGV and AMR systems from different vendors to interoperate with a common fleet management interface, eliminating proprietary communication protocols as a barrier to multi-vendor fleets.

<Info>
  VDA 5050 Version 2.0 introduced significant extensions including the `connection` message for broker health monitoring, `factsheet` messages for vehicle capability declaration, and enhanced action parameter typing. Always specify the VDA 5050 version number in the architecture — breaking changes between minor versions affect field-level compatibility.
</Info>

## Fleet manager deployment pattern

```text
Site automation hierarchy:

  ERP / WMS (SAP, Oracle, etc.)
       │  REST/OPC-UA order interface
       ▼
  Fleet Manager (FMS)
       │  VDA 5050 / MQTT
       ▼
  MQTT Broker (Mosquitto / HiveMQ / EMQX)
       │  per-vehicle topics
       ▼
  AMR Fleet (Wi-Fi / 5G private network)
  AMR-001 ... AMR-N
```

<Tip>
  Specify the MQTT QoS level for each message type explicitly: QoS 0 (at most once) is acceptable for visualization messages, QoS 1 (at least once) is required for order and state messages, and QoS 2 (exactly once) should be evaluated for instantAction safety-critical commands to prevent duplicate command execution.
</Tip>

<Warning>
  The fleet manager is a non-safety-rated system in most architectures — safety functions (e-stop, speed limiting, collision avoidance) are implemented on the AMR safety controller, not the fleet manager. The fleet manager can initiate a soft stop but cannot be the sole means of implementing a safety function.
</Warning>
