title: "The skill chain: how artifacts flow between skills"
description: "How builder xlsx outputs become inputs to downstream builders across the Foundation, Cobot, AMR, Cell+Ops, ROS2, V&V, AI/ML, and Cybersecurity lanes."
---

The chain is the core architectural feature of the Robotics Skills Suite. Every builder skill produces an xlsx workbook with named, versioned tabs. Downstream builders read from those named tabs as their inputs вҖ” meaning a change to a value in an upstream workbook automatically propagates to every skill that consumes it, without any manual copy-paste. This is what makes the suite more than a collection of templates: each phase is a node in a directed graph of structured file contracts, and the graph enforces data consistency across the entire project lifecycle.

## How the chain works

When you run a builder skill, it writes its output to a named xlsx file. Each tab in that file has a stable name вҖ” for example, `RiskRegister` or `PLr_Determination`. Downstream builder skills are configured to read from those specific tab names. If you update a hazard severity in the ISO 12100 Risk Assessment's `RiskRegister` tab, then re-run the ISO 13849-1 PLr builder, it reads the updated severity and recomputes the required Performance Level. Run the ISO 10218-1/2 Compliance Matrix builder next and it reads the new PLr value. Run the Declaration of Conformity builder last and it pulls from the compliance matrix.

<Note>
  Tab names in builder xlsx outputs are stable contracts. Do not rename, delete, or reorder tabs in a builder output. Downstream skills locate data by tab name, not by tab position. Renaming a tab will cause downstream builders to fail to read the expected data.
</Note>

## Chain execution order

<Steps>
  <Step title="Run the Foundation chain first">
    Always start with the ISO 12100 Risk Assessment. It is the root of the main chain and the source of hazard severity, frequency, and avoidability values that every downstream safety skill consumes.

    ```text
    ISO 12100 Risk Assessment
    вҶ’ Machinery Safety Lifecycle Plan
    вҶ’ Robot Cell Scope
    вҶ’ ISO 13849-1 PLr / IEC 62061 SIL
    вҶ’ ISO 10218-1/2 Compliance Matrix
    вҶ’ Declaration of Conformity (CE)
    ```
  </Step>
  <Step title="Branch into robot-type lanes">
    After the Foundation chain, branch into the lane that matches your robot type. Cobot, AMR, and Cell+Ops lanes each read from the Foundation xlsx outputs.
  </Step>
  <Step title="Run software and V&V lanes in parallel">
    The ROS2, V&V, AI/ML, and Cybersecurity lanes are largely independent of each other. Once the Foundation chain is complete, these lanes can run in parallel.
  </Step>
  <Step title="Run reviewer skills at each milestone">
    After completing a lane or reaching a customer review gate, run the reviewer skill for each builder output. Reviewer dashboards are independent artifacts вҖ” they do not affect the xlsx contracts or downstream chain execution.
  </Step>
</Steps>

## How upstream changes propagate

The chain propagates changes automatically because downstream builders read from named tabs at execution time вҖ” not at install time. Here is how a typical upstream change flows:

<Steps>
  <Step title="Edit the upstream xlsx">
    Open the ISO 12100 Risk Assessment workbook. Change a hazard severity from S1 (slight injury) to S2 (serious injury) for the pinch-point hazard on the cell entry.
  </Step>
  <Step title="Re-run the affected downstream builder">
    Trigger the ISO 13849-1 PLr builder, pointing it at the updated ISO 12100 xlsx. The builder reads the new severity value from the `RiskRegister` tab and recomputes the required Performance Level вҖ” in this case, the pinch-point function may move from PLc to PLd.
  </Step>
  <Step title="Cascade through the remaining chain">
    Re-run the ISO 10218-1/2 Compliance Matrix builder. It reads the new PLd requirement from the PLr workbook and flags any safety functions that were previously specified to only PLc as non-compliant. Re-run the Declaration of Conformity builder last to update the CE file.
  </Step>
  <Step title="Run reviewers to confirm">
    Run the reviewer skill for each updated workbook. The reviewer dashboards will show the updated compliance status and surface any new gaps introduced by the changed requirement.
  </Step>
</Steps>

<Info>
  You only need to re-run the skills that are downstream of the change. Skills upstream of the change are unaffected. If you change only the PLr xlsx вҖ” for example, to update an architecture decision вҖ” you only need to re-run the Compliance Matrix and Declaration of Conformity builders.
</Info>

## Stable xlsx contracts

The chain depends on tab names remaining stable. The suite enforces the following conventions across all builder outputs:

- Tab names use `PascalCase` with no spaces вҖ” for example, `RiskRegister`, `PLr_Determination`, `ComplianceMatrix`
- The `Metadata` tab is always present and always the last tab; it records skill version, generation timestamp, and input parameters
- Downstream builders read data ranges by named tab and named column header, not by cell address вҖ” column reordering within a tab is safe, but column renaming is not

<Warning>
  Never rename, delete, or merge tabs in a builder xlsx output. If you need to add notes or annotations, use a separate tab prefixed with `Notes_` вҖ” the convention signals to downstream builders that the tab is not a contract tab and can be safely ignored.
</Warning>
